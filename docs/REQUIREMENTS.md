# [문서 2] 요구사항 명세서 (SRS) (최종본)

## 2.1. 기능적 요구사항 (Functional Requirements)

### FR-A: Track A (공정한 평가 시스템)

| ID | 요구사항 명 | 상세 설명 | 근거 |
| :--- | :--- | :--- | :--- |
| **FR-A-1** | **조직/인원 관리** | | |
| FR-A-1.1 | 조직도 일괄 동기화 | 시스템이 정의하는 표준 형식(JSON/CSV)의 파일을 업로드하여, 연구소/센터-실-팀의 3단계 조직도를 일괄 동기화(생성, 수정, 삭제)할 수 있어야 한다. | |
| FR-A-1.2 | 인원 배정 (일괄) | 직원을 각 조직에 배정하고 입사/퇴사/이동 처리를 할 수 있어야 한다. (일괄 동기화에 포함) | |
| FR-A-1.3 | (수정됨) 멤버 중심의 프로젝트 참여 비중 설정 | - 실장은 평가 기간 내 소속 인원 **한 명을 기준으로**, 해당 인원이 참여하는 모든 프로젝트의 참여 비중(%)을 설정할 수 있어야 한다.<br>- 특정 인원의 모든 프로젝트 참여 비중의 **총합은 반드시 100%**여야 한다.<br>- 백엔드 API는 참여 비중의 총합이 100%가 아닌 경우, 요청을 거부하고 에러를 반환하여 데이터 정합성을 보장해야 한다.<br>- 이를 위해 다음의 "멤버 중심" API가 제공되어야 한다:<br>  - `GET /api/v1/users/{user_id}/project-weights`: 특정 사용자의 모든 프로젝트 참여 비중을 조회<br>  - `PUT /api/v1/users/{user_id}/project-weights`: 특정 사용자의 모든 프로젝트 참여 비중을 한 번에 업데이트 | (사용자 확인),<br>(프론트엔드 요구사항 구체화) |
| **FR-A-1.4** | **(추가됨) 개별 조직 관리** | **관리자는 개별 조직(연구소/센터, 실, 팀)을 생성, 수정, 삭제할 수 있어야 한다.** | (사용자 요청) |
| **FR-A-1.5** | **(추가됨) 개별 인원 관리** | **관리자는 개별 인원을 시스템에 추가, 수정, 삭제할 수 있어야 한다.** | (사용자 요청) |
| **FR-A-1.6** | **(추가됨) 개별 인원 배정** | **관리자는 개별 인원을 특정 조직에 배정하거나 다른 조직으로 이동시킬 수 있어야 한다.** | (사용자 요청) |
| **FR-A-1.7** | **(추가됨) 조직도/사용자 일괄 동기화 (JSON)** | **관리자는 시스템이 정의하는 특정 JSON 형식의 파일을 업로드하여, 전체 조직도(계층 구조 포함)와 사용자 정보(직급, 역할 포함)를 일괄적으로 동기화(생성, 수정)할 수 있어야 한다.** | (사용자 요청) |
| **FR-A-2** | **평가 설정** | | |
| FR-A-2.1 | 평가 주기 설정 | 평가는 연 2회(상반기/하반기) 설정된 기간에만 활성화되어야 한다. | |
| FR-A-2.2 | 등급 비율 설정 | 부서(실)가 받은 평가(S/A/B)에 따라 S/A 등급 부여 가능 인원 비율(60/40/20%)이 시스템에 설정되어야 한다. | |
| **FR-A-2.3** | **(추가됨) 평가 항목 가중치 설정** | **시스템 관리자(인사관리자)는 직책별 평가 항목(PM동료평가, PM평가, 정성평가 등)의 가중치(%)를 조정할 수 있는 관리자 페이지를 제공해야 한다.** | (사용자 요청) |
| **FR-A-3** | **평가 진행** | | |
| FR-A-3.1 | PM동료평가 | 피평가자는 자신을 제외한 프로젝트 동료에게 점수를 부여한다. <br> 부여하는 점수의 평균은 70점을 초과할 수 없으며, 초과 시 시스템은 제출을 제한하고 사용자에게 알림을 표시해야 한다. | |
| **FR-A-3.2** | **(수정됨) 동료평가 피드백 열람** | 동료평가 시 작성한 서술형 피드백은 **실장 및 상위 관리자, 인사관리자만** 열람 가능해야 한다. (피평가자 본인 열람 불가) |, (사용자 요청) |
| FR-A-3.3 | PM평가 | PM은 프로젝트 구성원에게 100점 만점의 점수를 부여한다. | |
| FR-A-3.4 | PM 본인 평가 | PM 직책 보유자의 'PM평가' 점수는 관리자(연구소장)가 오프라인 결과값을 시스템에 직접 입력할 수 있어야 한다. |, (사용자 확인) |
| FR-A-3.5 | 정성평가 | 팀장/실장은 피평가자에게 100점 만점의 정성평가 점수를 부여한다. | |
| **FR-A-4** | **등급 산출 및 조정** | | |
| FR-A-4.1 | 가중치 합산 | 직책별(일반직원, PM 등)로 **관리자가 설정한(FR-A-2.3)** 가중치에 따라 최종 점수를 자동 산출한다. |, (사용자 요청) |
| FR-A-4.2 | 복수 프로젝트 점수 합산 | 2개 이상 프로젝트 참여자는 (A프로젝트 점수 \* A비중%) \+ (B프로젝트 점수 \* B비중%) 공식으로 점수를 합산한다. | |
| FR-A-4.3 | 1차 등급 부여 | 실장은 산출된 점수 순서대로 배정된 S, A 등급을 부여한다. S/A/C/D 외 인원은 'B등급 대상자'가 된다. | |
| FR-A-4.4 | B+/B- 등급 조정 | 실장은 'B등급 대상자' 중 점수와 무관하게 B+와 B-를 부여할 인원을 선택한다. <br> **(제약조건)** 실 내 B+ 인원수와 B- 인원수는 반드시 동일해야 한다. | |
| FR-A-4.5 | 동점자 처리 | 등급 경계의 동점자 발생 시, 실장이 상위 등급 부여 대상자를 직접 선택(Check)할 수 있는 UI를 제공해야 한다. <br> **(제약조건)** 선택 가능한 인원수는 해당 등급에 배정된 잔여 TO(인원)를 초과할 수 없도록 시스템이 제한해야 한다. | |
| **FR-A-4.6** | **(추가됨) Growth & Culture 리포트 조회** | 실장 및 상위 관리자는 정성평가(FR-A-3.5), 등급 조정(FR-A-4.4), 동점자 처리(FR-A-4.5) 수행 시, 피평가자의 **'Growth & Culture 리포트'를 읽기 전용 참고 자료로 조회**할 수 있어야 한다.<br> - 이 리포트에는 **'협업 네트워크 요약'(FR-B-2.4)**과 **'강점 프로필 요약'(FR-B-3.5)**이 포함된다.<br> - 이 리포트의 데이터는 점수 합산(FR-A-4.1)에 영향을 주지 않는다. | (사용자 요청) |
| **FR-A-5** | **데이터 열람 권한** | | |
| FR-A-5.1 | 피평가자 열람 | 피평가자는 자신의 최종 등급과 각 PM에게 받은 PM평가 점수만 열람할 수 있다. | |
| FR-A-5.2 | 피평가자 열람 불가 | 피평가자는 자신의 동료평가 점수(개별/합산), 정성평가 점수, 최종 총점, 석차를 절대 열람할 수 없다. | |
| **FR-A-5.3** | **(수정됨) 관리자 열람** | 실장은 소속원의 모든 점수(동료, PM, 정성, 총점), 석차, 동료평가 익명 피드백(FR-A-3.2)을 열람할 수 있다.<br> - 실장 및 상위 관리자는 **'Growth & Culture 리포트'(FR-A-4.6)**를 열람할 수 있다.<br> - 상위 관리자 및 인사관리자는 FR-A-3.2에 따라 피드백을 열람할 수 있다. |, (사용자 요청) |
| **FR-A-5.4** | **(추가됨) 이력 조회** | - 구성원은 반기 별로 자신의 프로젝트 진행 이력과 평가 결과를 조회할 수 있다.<br>- 보직자는 자신의 하위 구성원의 반기 별 프로젝트 진행 이력과 평가 결과를 조회할 수 있다. | (사용자 요청) |
| **FR-A-6** | **(추가됨) 프로젝트 관리** | | |
| **FR-A-6.1** | 프로젝트 생성 및 설정 | **실장 이상의 보직자는 프로젝트를 생성할 수 있어야 한다.**<br>- 프로젝트에는 담당 PM과 유효기간(시작일, 종료일)을 설정할 수 있어야 한다. | (사용자 요청) |
| **FR-A-6.2** | 프로젝트 멤버 배정 | **실장 이상의 보직자는 프로젝트에 조직원을 배정할 수 있어야 한다.** | (사용자 요청) |
| **FR-A-6.3** | 평가 시 유효기간 적용 | **평가 점수 산출 시, 평가 기간 내에 유효기간이 포함되는 프로젝트의 데이터만 반영되어야 한다.** | (사용자 요청) |

### FR-B: Track B (성장과 인정의 문화 플랫폼)

| ID | 요구사항 명 | 상세 설명 |
| :--- | :--- | :--- |
| **FR-B-1** | **AI 기반 '나의 성장 회고록'** | |
| FR-B-1.1 | 외부 계정 연동 | 사용자는 자신의 Jira, Bitbucket 계정을 시스템에 (읽기 전용으로) 연동할 수 있어야 한다. |
| FR-B-1.2 | 회고록 생성 | 사용자가 기간(기본값: 최근 6개월, 설정 가능)을 설정하고 '회고록 생성'을 요청하면, AI(LLM)가 연동된 계정의 활동(커밋, 티켓 처리 내역)을 요약하여 텍스트 초안을 생성한다. <br> - 기본 LLM은 GPT를 사용하되, 향후 다른 LLM으로 교체할 수 있는 확장성을 고려하여 설계한다. |
| FR-B-1.3 | **프라이버시 보장 (중요)** | 생성된 회고록은 **오직 사용자 본인만** 조회, 수정, 삭제할 수 있어야 하며, 타인(관리자 포함)이 절대 열람할 수 없어야 한다. **('Growth & Culture 리포트'에 포함되지 않음)** |
| **FR-B-2** | **'협업 네트워크' 시각화** | |
| FR-B-2.1 | 관계 데이터 수집 | 시스템은 Jira(댓글, 멘션), Bitbucket(PR 리뷰, 코멘트) 데이터를 분석하여 '누가, 누구와, 어떤 유형의' 상호작용을 했는지에 대한 관계 데이터만을 수집한다. <br> **(제약조건)** 댓글, 코멘트 등의 실제 내용은 절대 수집 및 저장하지 않는다. |
| FR-B-2.2 | 네트워크 시각화 (전체 공개) | 모든 사용자는 조직/프로젝트별 협업 관계도를 네트워크 그래프 형태로 시각화하여 조회할 수 있어야 한다. |
| FR-B-2.3 | 협업 중심 분석 (전체 공개) | '가장 많은 리뷰를 한 사람', '가장 많은 도움을 준 사람(멘션된 사람)' 등 관계 중심의 분석 데이터를 제공한다. |
| **FR-B-2.4** | **(추가됨) 협업 네트워크 요약 (관리자용)** | 시스템은 `FR-B-2.1`의 데이터를 기반으로 개인별 협업 요약(예: 타 부서 협업 횟수, 도움 응답률 상위%)을 생성하여 `FR-A-4.6` 리포트에 제공한다. |
| **FR-B-3** | **'칭찬 인박스' & '강점 프로필'** | |
| FR-B-3.1 | 익명 칭찬 전송 | 사용자는 타인을 선택하여 칭찬 메시지와 강점 해시태그(#해결사)를 전송할 수 있다. <br> 전송 시, 발신자는 '익명의 동물' (예: 익명의 고라니)로 자동 변환된다. |
| FR-B-3.2 | 칭찬 인박스 (개인 비공개) | 칭찬 메시지는 **오직 수신자 본인만** 조회할 수 있는 개인 '칭찬 인박스'로 전달된다. (공개 피드 X) |
| FR-B-3.3 | 강점 프로필 (전체 공개) | 수신자가 받은 강점 해시태그는 자동으로 집계되어, **모든 사용자에게 공개**되는 '강점 프로필'에 뱃지 형태로 표시된다. (예: 🏆 #해결사 x 8회) |
| FR-B-3.4 | 데이터 분리 (중요) | 칭찬 메시지 원문과 발신자 정보는 철저히 비공개 처리되며, 오직 집계된 해시태그 결과만 프로필에 연동된다. **('Growth & Culture 리포트'에 메시지 원문 포함 안 됨)** |
| **FR-B-3.5** | **(추가됨) 강점 프로필 요약 (관리자용)** | 시스템은 `FR-B-3.3`의 데이터를 기반으로 개인별 강점 뱃지 현황을 요약하여 `FR-A-4.6` 리포트에 제공한다. |
| **FR-B-3.6** | **(추가됨) 칭찬 횟수 제한 (어뷰징 방지)** | 사용자가 다른 특정 사용자에게 보낼 수 있는 칭찬의 횟수는 평가 주기(반기)당 일정 횟수(예: 5회)로 제한된다. <br> - 이 횟수는 관리자가 설정 페이지에서 조정할 수 있어야 한다. | (사용자 요청) |

## 2.2. 비기능적 요구사항 (Non-Functional Requirements)

* **NFR-1: 보안 및 접근 제어 (Security & RBAC)**
    * `Track A`의 모든 데이터는 `FR-A-5`의 열람 권한 정책을 엄격하게 준수해야 한다. 인사관리자, 상위 관리자, 실장, 팀장, 일반사원의 역할 기반 접근 제어(RBAC)가 필수적이다.
* **NFR-2: (수정됨) 조건부 데이터 참조 (Conditional Data Reference)**
    * `Track B`의 집계 데이터(`FR-B-2.4`, `FR-B-3.5`)는 `FR-A-4.6`에서 정의된 대로 관리자에게 '읽기 전용' 참고 자료로만 제공될 수 있다.
    * `Track B`의 민감 데이터(`FR-B-1.3`, `FR-B-3.2`, 칭찬 발신자 정보)는 관리자를 포함한 그 누구에게도 절대 노출되어서는 안 된다.
* **NFR-3: 사용성 (Usability)**
    * `Track A`는 정확하고 오류 없는 데이터 입력/조회 UI를 제공해야 한다.
    * `Track B`는 재미있고(게이미피케이션), 빠르고, 부담 없이(익명성) 사용할 수 있는 UI/UX를 제공해야 한다.
* **NFR-4: 외부 시스템 연동 (Integration)**
    * `Track B` 기능을 위해 Jira 및 GitHub API에 대한 읽기 전용 연동이 필요하다.
* **NFR-5: 프라이버시 및 어뷰징 방지 (Privacy & Abuse Prevention)**
    * `Track B`의 익명 칭찬 발신자는 시스템 관리자조차 식별할 수 없도록 설계하는 것을 권장한다. (예: 발신자 정보 비저장)
